Partie 1 : Concept

ACID  

Atomique : tout ou rien, une modification des données doit être réalisée dans son intégralité ou pas du tout.


Cohérent : les données doivent toujours être cohérentes entre elles, même en cas d'erreur. Dans ce cas là, on effectuera un RollBack.


Isolée : Pas d'interférences entre les transactions. Utilisation des verrous et des points de synchronisation.


Durable : Lorsqu'une transaction s'est achevée, avec succès (Commit) ou en erreur (Rollback), les données doivent être dans un état stable et cohérent.

BASE  

Ba sic Availability : le sytème doit toujours être accessible.



Soft state : l'état de la base de données n'est pas garanti à un instant t.



Eventually Consistent : la cohérence des données à un instant t n'est pas primordiale.

 Notre cas d'apprentissage : MongoDB 

Pour Mongo nous allons parler de collections et de documents. Une collection est l'équivalent d'une table SQL. Et un document est l'équivalent d'un enregistrement.

Mongo utilise comme clef principale le champ _id  des documents. C'est le seul champ qui est commun à tous les documents de n'importe quelle collection. Il est également automatiquement indexé et unique. Son type par défaut est un ObjectId.

Un ObjectId est constitué de 12 octets (hexa) construit comme suit :
- 4 octets (timestamp) représentant la date de création de l'objet en secondes depuis Unix Epoch
- 5 octets (random) représentant une valeur aléatoire
- 3 octets (counter) représentant la valeur d'un compteur (initialisé aléatoirement)

Partie 2 : Modélisation

NoSQL règles de schéma pour les bases de données documentaires

La première chose à savoir c'est qu'il n'y a aucune règle qui permettent de définir un schéma de ba​se documentaire de manière général.

Le schéma doit être designé pour répondre aux besoins de l'application, il n'y a pas de formes normales à suivre comme on peut avoir en SQL.

Il existe néanmoins quelques patrons de conception qui permettent de répondre aux problèmatiques les plus courantes auxquelles une application doit faire face.

Nous en aborderons quelques uns dans ce chapitre du cours.

L'incorporation

Points positifs    :
 - Récupère toutes les données en une seule requête
 - Pas de perte de temps à utiliser des opérateurs de jointures
 - Permet également de mettre à jour toutes les données en une seule requête





Points négatifs   :
 - Temps de traitements plus long avec les gros documents
 - Taille limite à ne pas dépasser de 16Mo

La dénormalisation

Points positifs    :
 - Récupère toutes les données en une seule requête
 - Pas de perte de temps à utiliser des opérateurs de jointures
 - La donnée deromalisée peut avoir son propre cycle de vie
 




Points négatifs   :
 - Temps de traitements plus long avec les gros documents
 - Taille limite à ne pas dépasser de 16Mo
 - Peut déclencher la mise à jour en masse de documents selon le modèle choisi

Le référencement

Points positifs    :
 - Petits documents (16Mo limit très loin)
 - Pas de duplication de données
 - Rend possible de ne pas récupérer tous les liens si ils ne sont pas necessaires






Points négatifs   :
 - Besoin de faire beaucoup de jointures  

Les différents types de relations

One - to - One

C'est la relation la plus simple !

Il suffit de la gérer par un couple clef/valeur.

Par exemple ici la relation entre notre utilisateur et son employeur est gérée par le champ company dans lequel on a simplement renseigné la valeur MongoDB.

Dans certains cas si une des deux entités contient beaucoup de champs et qu'ils doivent souvent être récupérés en même temps que l'autre entité, alors on peut aussi utiliser un référencement voir une dénormalisation selon la fréquence de mise à jour.

Les différents types de relations

One - to - Few

Ici il faut comprendre le terme few comme not so many. 
C'est à dire pas des centaines ou des milliers de documents.

Dans ce cas-ci en général la meilleure solution est l'incorporation si la donnée de votre entité représentée par le terme few  ne necessite pas de cycle de vie propre.

Dans le cas contraire une dénormalisation des champs utiles (et uniquement ceux-ci) peut être réalisée.

Le référencement n'est en revanche pas une solution que l'on prefèrera choisir car cela génèrerait des jointures qui peuvent être évitées.

Les différents types de relations

One - to - Many

Dans ce genre de cas, c'est le référencement qu'il faut mettre en place surtout si les données référencées ont leur propre cycle de vie.

Le référencement est également un bon choix de conception si les données liées ne sont pas toujours necessaires lorsqu'on récupère notre document.


Les différents types de relations

One - to - Zillions

Dans ce genre de cas, c'est le référencement inversé qui est le plus optimisé. 

Ce sont les documents "zillions" qui conservent chacun une référence vers le document principal.

Les différents types de relations

Many - to - Many

On utilise le référencement pour gérer les associations multiples bi-directionnelles. 
Il faut cependant bien réfléchir à la manière dont vous avez besoin de récupérer la donnée.

Ici pour notre exemple, les cas d'usages couverts sont :
- La récupération de toutes les tâches d'une personne (via le tableau tasks de la collection Person)
- La récupération du propriétaire d'une tâche (via le champ owner de la collection Tasks)
-  La récupération de toutes les personnes assignées à une tâche (tous les documents de la collection Person qui ont l'id de la tâche dans leur tableau tasks)

Les différents types de relations

Ce qu'il faut retenir :

Quand vous utilisez l'inclusion, assurez vous de toujours la positionner dans l'entité parmi les deux qui est la plus requêtée.
Si vous utilisez de l'inclusion/dénormalisation au sein d'un tableau, assurez vous que la taille de ce tableau de grandisse pas au fil du temps (reste dans le même ordre de grandeur). Par exemple on ne stocke pas un tableau de commandes avec cette technique dans un document représentant un client.


Quand vous utilisez le référencement, gardez bien à l'esprit que seule votre application va pouvoir vous garantir l'intégrité de vos données.
MongoDB n'a pas de clefs étrangères, on ne peut pas supprimer des documents en cascade via une unique requête. Si vous choisissez de supprimer une référence, c'est à vous d'aller mettre à jour les documents qui maintenaient une référence dessus.
Si vous supprimez souvent des références, vous devriez considérer d'inverser votre référencement.

Les différents types de relations

Comment gérer au mieux les relations ?

Choisissez l'inclusion par défaut, sauf si vous avez une bonne raison de ne pas le faire.
Cela peut être le besoin d'accéder directement à cette entité fréquement ou alors le nombres de relations qu'elle porte. => Dénormalisation

Évitez les jointures !!!! Un opérateur existe pour ça mais il ne faut pas en abuser, son usage doit rester exceptionnel et occasionnel.

Les tableaux doivent toujours avoir une taille maitrisée, par exemple on peut considérer que le nombre de voitures que possèdent une personne va toujours rester dans le même ordre de grandeur. En revanche si ce sont des valeurs qui ne font que croître avec le temps (type commande) le tableau n'est pas la meilleure des conceptions.

Souvenez vous toujours que vous concevez votre base de données à partir des besoins de votre application, et non l'inverse !

Les patrons de conception

Pourquoi les utiliser ?

Les patrons de conception sont là pour vous aidez à construire votre modèle, ils vont vous permettre de mieux relier les différents cas business que doit couvrir votre application avec une modélisation de base de données. Ils sont également là pour vous aidez lorsque vous rencontrez des problèmes de performances important dans votre application.

Quels sont-ils ?

Il en existe un certain nombre, les plus populaires sont les suivants :

  • Computed Pattern
  • Bucket Pattern
  • Schema Versionning Pattern
  • Treen Pattern(s) (il en existe 4 différents)
  • Polymorphic Pattern
  • Approximation Pattern
  • Outlier Pattern

Computed Pattern

Pour quel problème ?

Opérations mathématiques et les opérations de type Fan Out & Roll Up (manipulation de données)
Pour des opérations très fréquentes qui produisent le même résultat sur les même données (avec beaucoup plus de lecture que d'écriture)


Quelle solution ?

Faire les calculs et les manipulations de données qui étaient réalisées à la lecture au moment de l'écriture des nouveaux documents.
Garder tous les documents de base dans une collection dédiée si un calcul doit être refait.

Cas d'usages courants :

  • IoT
  • Event Sourcing
  • Timeseries Data

Exemple Fan Out (une opération qui peut effectuer une ou plusieurs tâches pour retourner un résultat logique)

Computed Pattern

Exemple Roll Up (Opération qui consiste à regrouper des données ensemble  Drill Down)

Ce patron est très utilisé dans les applications de reporting pour faire des résumés des activités à la journée, à la semaine, au mois, ...

Computed Pattern

Points positifs    :
 - De très bonnes performances sur les requêtes en lecture
 - Peut économiser des ressources CPU et Disque





Points négatifs   :
 - Il peut être compliqué de voir le besoin qui necessite l'utilisation de ce patron
 - Un mauvais usage peut nuire à votre application, il peut entraîner des problèmes de performances

Bucket Pattern

Pour quel problème ?

Un très grand nombre de documents qui ne fait qu'augmenter au fil du temps (un historique, des données télématiques, ...).
Des documents qui pourraient dépasser la limite des 16Mo de Mongo à force d'ajouts fréquents de données.
Une relation One-To-Zillions.


Quelle solution ?

  1. Trouver la maille la plus fine de l'information dont vous avez besoin
  2. La stocker sous forme de tableau dans un document principal
  3. La relation One-To-Zillions se retrouve transformée et le document principal se retrouvera duppliqué en N documents qui contiendront chacun un tableau avec une quantité bornée de relations imbriquées

Cas d'usages courants :

  • IoT (données de capteurs)
  • Entrepôts de données (Data Warehouses)

Bucket Pattern

Cas d'usage IOT

imaginons 10 Millions de télpéhones qui envoient la valeur de leur batterie toutes les minutes, ça représente 600 Millions de données par heure.

Tenter de stocker chaque donnée dans un document séparé va très vite conduire à gérer énormément de documents (des milliards et des milliards)

Si on tentait de regrouper toutes les mesures d'un même téléphone dans un seul document au sein d'un tableau, on finirait par arriver à la limite de 16Mo qu'impose Mongo pour ses documents. De manière générale, MongoDB est plus performant en traitant de petits documents bien indexés que de gros documents.

Cas d'usage IOT : Utilisation du patron

La granularité des données stockées doit provenir des besoins de l'application

Bucket Pattern

Points positifs    :
 - Permet de contrôler la taille des documents retournés et manipulés
 - Permet de factoriser/simplifier les données





Points négatifs   :
 - Peut nuire aux performances si la taille du Bucket n'est pas définie en fonction des besoins de l'application
 - Pas très user friendly comme structure, le modèle n'est pas simple à utiliser pour de la BI ou de la Data Science

Schema Versioning Pattern

Pour quel problème ?

Gérer les migrations de schéma de vos données lorsque vous faites du BigData. Les migrations peuvent prendre des heures, voir même des semaines.


Quelle solution ?

  • Chaque document contient un champ contenant sa version actuelle du schéma
  • Faire en sorte que l'application puisse gérer les documents de tous les schémas présents en base
  • Choisir la stratégie de migration => One Shot, au fil de l'eau, par paquets, ...

Cas d'usages courants :

  • Toutes les applications qui ont une très faible tolérance à l'arrêt.

Schema Versioning Pattern

Points positifs    :
 - Permet de migrer des données sans rupture de service
 - Rend les migrations de données moins stressantes pour la base de donnée
 - Aide à réduire la dette technique (en permettant par exemple de supprimer les champs qui ne sont plus utilisés en tâche de fond de l'application)



Points négatifs   :
 - Les indexs définis sur les champs à migrer vont devoir vivre en double le temps que la migration s'effectue (utilise plus d'espace disque et de RAM)

Polymorphic Pattern

Pour quel problème ?

Des objets sont différents mais ils sont plus semblables que différents et il est impératif qu'ils restent dans la même collection.


Quelle solution ?

  • Chaque document contient un champ contenant l'information de quel type d'objet il s'agit
  • Faire en sorte que l'application puisse gérer les différentes structures de documents présentes

Cas d'usages courants :

  • Catalogue produits / Application de vente en ligne

Polymorphic Pattern

Points positifs    :
 - Facile à implémenter
 - Rend possible des requêtes d'aggregation complexe sur une seule collection (plus performant que de requêter plusieurs collections)




Points négatifs   :
 - Au fur et à mesure du temps qui passe et des évolutions que l'on fait sur l'application, cela peut pousser vers des documents qui deviendraient plus différents que semblables ce qui rendrait la collection plus difficile à utiliser.

Partie 3 : Optimisation

Indexation

Les indexs sont là pour éviter les problèmes de performances (requêtes lentes) d'une application.

Lorsque l'on manipule des bases NoSQL on a toujours besoin d'indexs (99% du temps). Ces bases sont faites principalement pour gérer du BigData et donc devoir éxecuter des requêtes parmi des millions, des milliards voir plus encore de documents. 

Les indexs MongoDB sont des arbres binaires B-tree.

Indexation

Les gains de performances que l'on obtient en ajoutant des indexs ne sont pas gratuits. Pour chaque index sur un collection, on va réduire la vitesse d'ajout d'un document au sein de cette collection. 

Il faut s'assurer que les indexs que l'on ajoute ne sont pas inutiles, qu'ils conduisent bien à un gain de performance sur l'application.
Lors de la création d'un index vous pouvez lui spécifier certaines options (partiel, TTL, unique, insensible à la casse).

Il existe différents types d'indexs : 

  • Single field
  • Compound
  • Multikey
  • Partial
  • TTL
  • ...

Single field indexes

Il y'a au moins un index de ce type sur chacune des collections, l'index présent sur le champ '_id' par défaut.

Un index de ce type ne comporte qu'un seul champ (il peut être descendant ou ascendant peu importe). 
Ce champ peut aussi bien se trouver à la racide du document ou dans un sous-document.

db.collection.createIndex({score : 1})

db.collection.createIndex({score : -1})

db.collection.createIndex({location.state : 1})

Ces indexs sont utilisés lorsque l'on fait des requêtes pour rechercher une valeur ($eq), une plage de valeurs ($gte, $gt, $lte, $lt), ou parmi une liste de valeurs ($in)

Compound indexes

Un index de ce type comporte au moins deux champs (racine ou sous-documents). Pour chaque champ, il faut spécifier si il doit être ascendant ou descendant. 

db.collection.createIndex({score : 1, location.State : 1})

db.collection.createIndex({score : -1, location.State : 1})

Ces indexs sont utilisés lorsque l'on fait des requêtes pour rechercher une valeur ($eq), une plage de valeurs ($gte, $gt, $lte, $lt), ou parmi une liste de valeurs ($in) tout comme les indexs simples.

Multikey indexes

C'est un index qui est sur un ou plusieurs champ d'un même tableau.
Ils sont plus gourmands car pour chaque objet dans le tableau indexé, on va avoir une clef d'index distincte. Si vous indexez un tableau de 5 éléments vous allez ajouter 5 clefs à votre index. 

Il n'est pas possible d'indexer plus d'un tableau dans le même index car cela reviendrait à faire un produit cartésien pour chaque combinaison possible. C'est interdit car cela entrainerait une création beaucoup trop importante de clefs et ferait chuter les performances immédiatement.

db.collection.createIndex({locations.state : 1})

il est recommandé d'éviter d'utiliser des indexs sur des tableaux, cela peut paraître pratique à première vue, mais si votre tableau n'est pas issu d'une relation One-To-Few, mais d'une Many ou Zillion, votre index sera tellement volumineux qu'il risque de ne pas tenir en RAM lorsque vous tenterez de l'utiliser dans une requête. Ce qui va drastiquement réduire votre gain de performance.

Partial indexes

C'est une particularité que l'on peut/doit ajouter aux indexs vus précédement. 

Il est souvent avantageux de ne pas créer un index sur toute une collection mais sur un sous-ensemble de ses documents. Cela va réduire la taille de l'index et réduire son coût de création. Cela n'aura également pas d'incidence sur les performances d'insertion des documents ne faisant pas partie de ce sous-ensemble.

Par exemple, si vous avez certains champs optionnels dans vos documents que vous voulez indexer, il est interessant d'utiliser un index partiel pour ne cibler que les documents contenant le champ que vous souhaitez indexer.

Partial indexes

Attention cependant à leur utilisation, ces indexs sont partiels comme leur nom l'indique.

Ils ne référencent donc pas tous les documents de la collection !

Admettons qu'un index partiel ait été créé avec la condition suivante :

db.collection.createIndex({ score : 1, partialFilterExpression: {"score":{exists : true}} })
Si vous faites une requête demandant tous les documents qui n'ont pas de score, vous n'utiliserez pas l'index !

db.collection.createIndex({ score : 1, partialFilterExpression: {"score":{$gt : 7}} })
Si vous demandez tous les documents qui ont un score compris entre 5 et 10, vous n'utiliserez pas l'index !

Time To Live indexes

Tout comme les partialExpressions, le TTL est une option que l'on peut ajouter à un index. 
Cela permet de garder le contrôle sur le nombre de documents d'une collection de manière automatique en fonction du temps.
 On ne peut pas toujours mettre un TTL sur une collection, il faut que votre besoin business accepte des données limitées dans le temps.

Un TTL se fixe sur un champ de type date , il permet de supprimer des documents au bout d'un certain nombre de secondes passées ou même de supprimer des documents à une date fixe donnée.

db.log_events.createIndex( { "createdAt": 1 }, { expireAfterSeconds: 3600 } )
Dans ce premier cas MongoDB supprimera les documents une heure après leur date de création.

db.log_events.createIndex( { "expireAt": 1 }, { expireAfterSeconds: 0 } )
Dans ce deuxième cas MongoDB supprimera les documents lorsque la date spécifiée dans le champ expireAt sera atteinte.

Trier avec un index

Il existe plusieurs manières de trier les documents dans MongoDB.

La première consiste à faire ce tri en utilisant la RAM du serveur de base de données en spécifiant dans la requête un ou plusieurs champs à trier (.sort() ou $sort). Par défaut Mongo ne permet pas de trier les documents en RAM si la taille totale dépasse les 32 Mo. Cette valeur peut être augmentée dans la configuration du serveur si vous en avez vraiment besoin mais ce n'est pas recommandé.

La deuxième à utiliser une commande d'agrégation. En effet dans les options d'une agrégation, on peut spécifier à MongoDB d'écrire les résultats temporaires sur le disque pour éviter de dépasser cette limite de RAM (option allowDiskUse : true).

La troisième c'est d'utiliser un index. L'ordre de tri d'un index permet de récupérer les documents dans le même ordre lorsque l'on exécute une requête.
db.collection.createIndex({score : 1})
Si vous exécutez la requête suivante : db.collection.find({score:{$gt : 5 }}) 
Vous allez récupérer tous les enregistrements qui ont un score supérieur à 5 triés du plus petit au plus grand.

Utiliser le(s) préfixe(s) d'un index

Un index composé contient au moins un préfixe. Prenons pour exemple l'index ci-dessous :
{pays : 1, region : 1, departement : 1, ville : 1}

Cet index composé contient trois autres indexs que l'on appelle ses préfixes.
{pays : 1, region : 1, departement : 1}
{pays : 1, region : 1}
{pays : 1}

Cependant {region : 1} n'est pas un index possible, ni {pays :1, departement :1}.

Lorsque vous créez des indexs composés, essayez de toujours mettre en premier les champs sur lesquels vous allez toujours baser votre requête et ensuite les champs optionnels dans l'ordre de leurs occurences. Cela permet de limiter grandement le nombre d'indexs à créer pour une collection.

Intersection d'indexs

Les intersections d'indexs ne s'effectuent qu'entre des indexs simples, composés et des préfixes d'index simple ou composé.

db.villes.find({pays : "FRANCE", nbHabitants : {$gt : 100 000}})

L'index composé répondant pleinement à cette requête est le suivant : {pays : 1, nbHabitants: 1}

Cependant si on reprend l'index du slide précédent : {pays : 1, region : 1, departement : 1, ville : 1}
On remarque que {pays : 1} est déjà un préfixe de cet index. 
Pour utiliser l'intersection d'index, il faudrait uniquement rajouter l'index : {nbHabitants : 1} ce qui serait moins gourmand en terme de stockage que d'ajouter un index composé complet pour cette requête.

Il faut garder en tête qu'une intersection d'index n'est pas aussi optimale que d'avoir un index composé, par contre cela peut s'avérer très utile dans certains cas, surtout si c'est pour des requêtes qui ne sont pas effectuées très souvent et qui peuvent avoir un temps de réponse moins bon.
